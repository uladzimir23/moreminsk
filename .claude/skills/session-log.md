# Skill: Session Log

## Когда создавать session log

В конце сессии, если были **реальные изменения**:

- Коммиты в git
- Создание/изменение файлов в `src/` или `docs/`
- Архитектурные решения (новый ADR)
- Принятые бизнес-решения

**Не создавать** при чисто исследовательских сессиях (только чтение, поиск) — если ничего не изменено.

## Где создавать

`docs/70 - Meetings & Logs/YYYY-MM-DD HH:mm Claude Session.md`

Формат имени — с датой и временем **начала** сессии (не текущей).

## Шаблон

```markdown
---
date: YYYY-MM-DD
time: HH:mm
type: session-log
topic: "Краткая тема 1 предложением"
tags: [session, claude-code, <модуль>]
---

# Сессия Claude Code — YYYY-MM-DD HH:mm

## Что обсуждали

- Пункт 1 (что решали)
- Пункт 2

## Решения принятые

- Решение 1 (краткая формулировка)
- Решение 2

## Изменения в коде

- `file/path.ts` — что изменилось
- Новые файлы: `...`
- Новые ADR: `...`
- Коммит: `hash (первые 7) — commit message`

## Следующие шаги

1. Что ещё нужно сделать
2. Вопросы, требующие ответа заказчика
3. Открытые задачи

## Ссылки

- [[../00 - Indexes/Dashboard]]
- [[соответствующий документ]]
```

## Шаблон-источник

`docs/80 - Templates/Claude Session.md` — копировать и заполнять.

## Dashboard-ссылка

После создания session log — добавить / не добавить ссылку в `docs/00 - Indexes/Dashboard.md` (если session важная).

## Ключевые события, которые всегда документируются

- Архитектурное решение (ADR создан) → раздел «Решения принятые»
- Коммит с `feat:` или `refactor:` → раздел «Изменения в коде»
- Запрос к заказчику (требует ответа) → раздел «Следующие шаги»

## Что НЕ писать в session log

- ❌ Дословный транскрипт разговора
- ❌ Код целиком (ссылки на файлы достаточно)
- ❌ Раздумывания и сомнения («я подумал что...»)
- ❌ «Помог пользователю с X» — это банально и не документирует ничего
