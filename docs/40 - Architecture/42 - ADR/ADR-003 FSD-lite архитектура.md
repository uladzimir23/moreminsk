# ADR-003 — FSD-lite архитектура

## Контекст

Проект — небольшой контентный сайт (~28 страниц, ~15 виджетов). Нужна структура, которая:
- Масштабируется до 50+ страниц и 30+ виджетов без хаоса
- Понятна дизайнеру/копирайтеру, если они лезут в код
- Совместима с Next.js App Router
- Соответствует подходу в других проектах исполнителя (neuro-center, flex-glass)

Классический FSD (Feature-Sliced Design) с 6 слоями — избыточен для лендинга.

## Решение

Берём **FSD-lite**: 5 слоёв, упрощённая структура внутри слоя.

```
app → widgets → features → entities → shared
```

- **app** — страницы Next.js (`src/app/[locale]/...`), layout'ы, sitemap
- **widgets** — крупные UI-блоки страниц (Hero, FleetGrid, FAQ...)
- **features** — интерактивные фичи (Booking wizard, YachtFilter, BookingCTA, ThemeToggle, LocaleSwitcher)
- **entities** — бизнес-сущности с карточками (Yacht, Service, Review)
- **shared** — атомарный UI, утилиты, типы, стили, контент, конфиг

## Правила

1. **Импорт только сверху вниз:** `app` может импортить `widgets`, `widgets` может импортить `features` и т.д. Наоборот — нельзя.
2. **Внутри слоя** — нет кросс-импортов между сиблингами. `widgets/hero` не импортит `widgets/footer`. Связь — через `shared` или `features`.
3. **Публичный API:** каждый модуль экспортирует через свой `index.ts` — внутренние файлы не видны снаружи.
4. **Сущность содержит:** `ui/` (карточка), `model/` (типы + логика), опционально `api/` (в MVP — нет).

## Последствия

### Позитивные

- Предсказуемое место для нового виджета / фичи / сущности.
- Лёгкий рефакторинг — нет циклических зависимостей.
- Тот же паттерн, что в neuro-center → easy onboarding.
- Стили локальны компоненту (`*.module.scss` рядом).

### Негативные

- Слой `entities` на мелком сайте может выглядеть избыточно — но лучше иметь и не использовать, чем потом переделывать.
- Новому разработчику нужно 10 минут на объяснение FSD.

### Нейтральные

- Не используем shared/ui как design-system-пакет — это просто папка. Выделять в отдельный npm-пакет смысла нет.

## Альтернативы (отвергнутые)

1. **Плоская структура `src/components`** — быстро разрастается в хаос при 30+ компонентах.
2. **Atomic Design** (atoms/molecules/organisms) — абстрактные категории, плохо ложится на лендинг-блоки.
3. **Чистая Feature-based** — без разделения на widgets/features/entities теряется семантика.
4. **Полный FSD с 6 слоями** — избыточно для лендинга.

## Реализация

- `tsconfig.json`: `paths` алиасы `@/widgets/*`, `@/features/*`, `@/entities/*`, `@/shared/*`, `@/app/*`
- ESLint правило `import/no-restricted-paths` (при желании) — запретить импорт снизу вверх
